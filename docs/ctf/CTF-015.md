## Deuda

1. **`backend/apps/finanzas/views.py:148-159` (`transferir-entre-cajas`):** el código hace
   `sesion.cajas.get(...)`, pero `SesionCajaFisica` **no tiene** relación `cajas` (la FK
   real es `cajas_virtuales` sobre `CajaFisica`). El `AttributeError` cae en el `except`
   de las líneas 150-151 y el endpoint responde **400 siempre** — es código muerto que
   aparenta validación. Las líneas 152-153, 156 y 159 son inalcanzables hoy (documentado
   como no-cubrible en el gate diff-cover de #101).
2. **Quirk relacionado:** la API permite crear órdenes de fabricación con `cantidad=0`
   (el costeo luego responde 400 para no dividir entre cero —
   `test_costeo_orden_cantidad_cero_devuelve_400`). Falta validación `cantidad > 0` en el
   serializer de OF.

## Por qué se DIFIERE (y no en este PR)

El PR #101 tiene regla dura de **no tocar código de app** (es el PR que repara el gate de
cobertura; mezclar un bugfix de negocio rompería R-PROC-2). Los tests de #101 documentan
el comportamiento actual con honestidad; este CTF le pone dueño y fecha para que el 400
fosilizado no se vuelva "comportamiento esperado" permanente.

## Condición de cierre

1. Corregir la relación usada por `transferir-entre-cajas` (o el modelo, según diseño) y
   reescribir los tests de `TestTransferirEntreCajasRoto` para exigir el flujo FELIZ
   (200 + efectos en BD) además de las ramas de error reales (líneas 152-159 cubiertas).
2. Agregar validación `cantidad > 0` al crear/actualizar OF + test (400 con mensaje claro).
3. Suite verde + diff-cover ≥95% se mantienen.

## Cierre (2026-06-12)

1. `transferir-entre-cajas` resuelve las cajas vía `sesion.caja_fisica.cajas_virtuales`
   (la relación real) con excepciones acotadas — camino feliz 200 con doble movimiento
   y saldos actualizados; 400 reales para caja ajena/inexistente/malformada, monto
   inválido y saldo insuficiente. Tests: `TestSesionCajaTransferirEntreCajas`
   (`tests_api/test_diffcov_finanzas_tesoreria.py`), reescritos para exigir el
   comportamiento correcto (el 400 fosilizado dejó de ser "esperado").
2. `OrdenProduccionSerializer.validate_cantidad` rechaza `cantidad <= 0` en create y
   update con "La cantidad de la orden debe ser positiva." (paridad con el service).
   Tests en `tests_api/test_diffcov_manufactura.py`; el costeo de una OF legada con
   cantidad 0 (simulada con `update()`) sigue respondiendo 400 controlado.
3. Mismo PR (bugs lote 3): reparado `SesionCajaFisicaViewSet.perform_create`
   (FieldError 500 por campo inexistente `es_fisica` + kwargs/método inexistentes +
   lookup sin scope de tenant).
