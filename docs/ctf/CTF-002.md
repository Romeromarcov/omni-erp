# CTF-002: DSL Runtime para Primitivas Declarativas (entidades/estados/reglas/vistas)

| Campo        | Valor                                           |
|--------------|-------------------------------------------------|
| **ID**       | CTF-002                                         |
| **Título**   | Activar runtime completo del DSL de personalización |
| **Regla**    | R-CODE-12 (runtime personalización)             |
| **Estado**   | CERRADO — 2026-05-24                            |
| **Fase**     | Fase 0 → Fase 1                                 |
| **Vence**    | 2026-08-01                                      |
| **Owner**    | equipo-plataforma                               |
| **ADR ref.** | ADR-005                                         |

## Deuda

El DSL de personalización (`apps/personalizacion/dsl.py`) soporta las
primitivas `campos` y `conectores` en modo PoC. Las primitivas
`entidades`, `estados`, `reglas` y `vistas` se registran en el
`config_dict` pero el runtime **no las aplica**: se emite una advertencia
(generada en `aplicar_config()`) en lugar de ejecutar la transformación.

## Impacto

- Los clientes no pueden personalizar entidades nuevas ni estados de flujo
  de trabajo sin modificar código Python.
- Los `ConfigAgente` que referencian `reglas` no se ejecutan.

## Condición de cierre

1. `aplicar_config()` procesa `entidades` creando tablas dinámicas o
   actualizando el esquema vía migrations generadas al vuelo (o un proxy).
2. `estados` aplica transiciones de estado a los modelos referenciados.
3. `reglas` evalúa condiciones y acciones descritas en el DSL.
4. `vistas` genera serializers y viewsets dinámicos.
5. Tests de integración cubren los 4 tipos de primitivas con al menos
   2 configuraciones distintas cada uno.

## Riesgo de prórroga

Sin este runtime el producto sólo soporta multi-tenant en datos, no en
schema. Los contratos SaaS con personalización quedan bloqueados.
