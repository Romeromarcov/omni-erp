| **Estado**   | ABIERTO — 2026-06-09                                        |
| **Fase**     | Plan 05 — Hardening, P0-1 (rollout RLS)                     |

## Deuda

La infraestructura de Row Level Security (P0-1) está mergeada y verificada
(`RLS_ENABLED=False` por defecto: sin cambio de runtime). Pero **el enforcement
de RLS depende del rol con que la app conecta a PostgreSQL**: un rol con
`SUPERUSER` o `BYPASSRLS` **salta las políticas aunque estén `FORCE`**.

Hoy la app conecta con el rol **dueño de las tablas** (mismo rol para migraciones
y runtime; ver `DATABASES` en `settings_base.py`). El enforcement web funciona
porque el middleware baja el bypass por request, pero el modelo es frágil:

- Cualquier path que toque la BD **sin** pasar por el middleware (una tarea
  Celery, un management command, un endpoint nuevo mal cableado) corre con el
  default de conexión `bypass='on'` → **sin aislamiento**.
- Si el rol de conexión en algún entorno es superusuario, RLS **no aplica en
  absoluto**. Esto se descubrió en CI: el Postgres de CI conecta como
  superusuario y **todos los tests de RLS fallaban** hasta que se introdujo un
  `SET ROLE` a un rol no-privilegiado en los tests (`conftest.rls_test_role`).

La forma correcta y robusta es la que hoy se simula en los tests con `SET ROLE`:
**un rol de aplicación dedicado, no-dueño de las tablas, sin `SUPERUSER` ni
`BYPASSRLS`**. Con eso el aislamiento es **fail-closed por construcción** (la app
nunca puede saltar RLS), las migraciones siguen corriendo como el rol dueño, y se
elimina la dependencia de que "todo path pase por el middleware".

## Condición de cierre

1. **Crear el rol de aplicación** `omni_app` (o equivalente) `NOSUPERUSER
   NOBYPASSRLS`, no-dueño de las tablas, con `GRANT` de `SELECT/INSERT/UPDATE/
   DELETE` sobre el esquema de datos y `USAGE`/`SELECT` sobre las secuencias.
   Migraciones y DDL siguen corriendo con el rol dueño (separado).
2. **Conectar el runtime con ese rol** (variable de entorno / `DATABASE_URL` del
   servicio web y de los workers) — distinto del rol de migraciones.
3. **Verificar fail-closed de verdad**: una conexión de la app sin contexto RLS
   (p. ej. una tarea Celery que no fije el contexto) **no ve filas** de tablas
   con RLS, en vez de verlas todas. Ajustar las tareas/comandos que legítimamente
   agregan cross-tenant para que usen `rls.rls_bypass()` o fijen el contexto
   explícitamente.
4. **CI sin el truco de `SET ROLE`**: configurar el job de backend para conectar
   con un rol no-superusuario, de modo que los tests de RLS verifiquen el
   enforcement con el mismo modelo que producción (y poder retirar la fixture
   `rls_test_role` cuando deje de ser necesaria).
5. **Activar `RLS_ENABLED=True`** primero en **staging** (validación end-to-end
   de los flujos críticos) y luego en producción, documentando el rollback.
6. Gate de cierre por PR (build, tests, `/security-review`, `/code-review`).

## Riesgo de prórroga

Mientras esto no se cierre, **RLS no puede activarse en producción** y el
aislamiento multi-tenant sigue dependiendo al 100 % del filtrado a nivel de
aplicación (`get_empresas_visible` + filtros de queryset) — sin la red de
seguridad de la base de datos que motivó P0-1. Un solo bug de filtrado seguiría
siendo una fuga cross-tenant sin defensa en profundidad.

## Notas

- Follow-ups menores asociados (no bloqueantes, ver P0-1 en el plan):
  memoización por request de `get_empresas_visible` en el middleware; documentar
  que con RLS activo el acceso de Django admin requiere `es_superusuario_omni`;
  fijar el contexto RLS dentro del generador SSE si se habilita
  `CONN_MAX_AGE`/pgbouncer en *session pooling*.
- Extender RLS al resto de las ~92 tablas multi-tenant (hoy 15 con RLS:
  piloto + lote 2) puede avanzar en paralelo con `RLS_ENABLED=False`; la
  activación real queda gobernada por este CTF.
