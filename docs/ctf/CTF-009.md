# CTF-009: Drift `es_superusuario_innova` (frontend) vs `es_superusuario_omni` (backend)

| Campo        | Valor                                                        |
|--------------|-------------------------------------------------------------|
| **ID**       | CTF-009                                                     |
| **Título**   | Alinear el nombre del rol de superusuario del proveedor      |
| **Regla**    | R-CODE (consistencia de contrato API ↔ frontend)            |
| **Estado**   | CERRADO — 2026-06-07                                        |
| **Fase**     | Previo a guards de plataforma/proveedor (Planes B y C)      |
| **Vence**    | 2026-07-15                                                  |

| **Ref.**     | `frontend/src/services/users.ts`, `backend/apps/core/models.py`, `frontend/src/api/schema.d.ts` |

## Deuda

El backend define el rol del dueño del software como **`es_superusuario_omni`**
(`apps/core/models.py`; aparece así en el OpenAPI `schema.d.ts`). El frontend, en cambio,
declara y consume **`es_superusuario_innova`** (`frontend/src/services/users.ts` y varias
páginas de Métodos de Pago / creación de usuario) — probable resto de un rebranding
"Innova" → "Omni".

Consecuencia: el campo del backend **no se está leyendo** en el frontend bajo el nombre
correcto. Cualquier guard de UI basado en "soy el proveedor del software" hoy está roto o
apoyado en un campo equivocado. Es **bloqueante de calidad** para el panel SaaS (Plan C) y
para los guards de plataforma (Plan B), donde se decide quién es el superusuario Omni.

## Por qué se documenta como CTF (y no se arregla inline ahora)

1. El cambio toca múltiples páginas del frontend y debe verificarse de forma aislada
   (no mezclarlo con la persistencia de planes ni con WIP de frontend).
2. Requiere confirmar que no haya un campo legacy `es_superusuario_innova` real en algún
   contrato antiguo antes de renombrar; barrido controlado.

## Condición de cierre

1. Unificar a `es_superusuario_omni` en frontend (interfaz `Usuario`, formularios, guards).
2. Regenerar tipos desde OpenAPI (`npm run gen:api-types`) y verificar drift cero.
3. Tests/render condicional del rol verdes; `npx tsc -b` y `npm run lint` limpios.
4. Confirmar que ningún endpoint sigue dependiendo del nombre viejo.

## Cierre — 2026-06-07

- Unificado a `es_superusuario_omni` en `frontend/src/services/users.ts` (interfaz `Usuario`),
  `pages/Core/Usuarios/UserCreatePage.tsx` y las 3 páginas de Métodos de Pago
  (`MetodoPagoListPage`/`DetailPage`/`CreatePage`).
- Bug colateral del rebranding corregido en backend: `apps/core/models.py`
  (`Dispositivo.puede_crear_caja_fisica`) leía `es_superusuario_innova` con default `False`,
  por lo que **siempre** devolvía `False` tras el rename del modelo. Ahora lee `es_superusuario_omni`.
- `schema.d.ts` (OpenAPI) ya exponía `es_superusuario_omni`; no hubo drift de tipos generados,
  el drift era solo en código escrito a mano. Barrido `grep` confirma cero ocurrencias vivas
  (solo quedan migraciones históricas `0001/0006/0007` y docs, que no se tocan).
- Gate: `npx tsc -b` y `eslint` limpios; `manage.py check` y `makemigrations --check` sin cambios.
