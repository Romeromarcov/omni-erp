# CTF-007: `picomatch` ≤2.3.1 (alta) en tooling de lint del frontend

| Campo        | Valor                                                          |
|--------------|----------------------------------------------------------------|
| **ID**       | CTF-007                                                        |
| **Título**   | CVE alta de `picomatch` vía `typescript-eslint` (build/lint)   |
| **Regla**    | A2/A0 (audits de dependencias High/Critical) — plan cero-dudas |
| **Estado**   | CERRADO — 2026-06-12                                           |
| **Fase**     | Fase 5 (gates de seguridad)                                    |
| **Vence**    | 2026-09-01                                                     |
| **Owner**    | equipo-frontend                                               |

## Deuda

`npm audit fix` cerró **11 de 12** vulnerabilidades del frontend (de 6 high + 6
moderate a **1 high**). La restante es `picomatch ≤2.3.1` (ReDoS + method-injection
en glob matching). Cadena única que la arrastra:

```
typescript-eslint → @typescript-eslint/typescript-estree → fast-glob@3.3.3
  → micromatch@4.0.8 → picomatch@2.3.1
```

Todo el resto del árbol ya usa `picomatch@4.0.4`.

## Por qué se DIFIERE (riesgo real bajo)

1. **Es dependencia de build/lint, no se empaqueta al navegador.** El glob de
   `picomatch` aquí sólo lo invoca `eslint` sobre los archivos del repo (entradas
   controladas por el equipo, no por un atacante). El vector ReDoS/inyección no es
   alcanzable en runtime de la app.
2. **No hay fix no-disruptivo.** `micromatch@4.0.8` fija `picomatch@^2`; forzar
   `overrides: picomatch@^4` mete la API 4.x dentro de `micromatch` 4 (incompatible)
   y puede romper el file-matching de eslint (job `lint` BLOQUEANTE). `npm audit fix
   --force` subiría parents a versiones major (riesgo de romper el build).
3. Se resuelve solo cuando `typescript-eslint`/`fast-glob` publiquen una versión que
   dependa de `picomatch@4`.

## Plan de cierre

- Revisar en cada bump de `typescript-eslint` si la cadena ya migró a `picomatch@4`.
- Mientras tanto, `npm audit` se mantiene **bloqueante a nivel `critical`** (0 critical);
  el job reporta el high pero no tumba el pipeline. Al cerrar este CTF se sube a
  `--audit-level=high` bloqueante y se añade `Security scan` a branch protection.

## Cierre (2026-06-12)

Se cerró por la vía prevista ("se resuelve cuando `typescript-eslint` publique
una versión sin la cadena"), sin necesidad de `overrides`:

1. **Upgrade directo** `typescript-eslint` `8.38.0 → 8.61.0` (mismo major,
   devDependency): `@typescript-eslint/typescript-estree@8.61` reemplazó
   `fast-glob` por `tinyglobby` + `minimatch`, así que la cadena
   `fast-glob → micromatch@4.0.8 → picomatch@2.3.1` **desapareció del árbol**.
   Todo el `picomatch` restante es `4.0.4`. (Alternativa descartada:
   `overrides` a `picomatch@2.3.2` — existe el parche dentro de `^2.3.1`,
   pero el upgrade lo hace innecesario y de paso moderniza el linter.)
2. **Verificado:** `npm audit --audit-level=high` → **0 vulnerabilidades**
   (también con `--omit=dev`); lint, tsc y la suite completa en verde con el
   linter nuevo.
3. **Gate activado** en `.github/workflows/ci.yml`:
   - Job `frontend`: paso nuevo **bloqueante** `npm audit --audit-level=high`
     tras el install. **Sin `--omit=dev`** a propósito: este mismo CTF
     rastreaba un HIGH de devDependencies; omitirlas habría ocultado el
     hallazgo, y el toolchain de build/lint (corre en CI y máquinas dev)
     también es superficie de supply-chain.
   - Job `security-scan`: `npm audit` sube de `--audit-level=critical` a
     `--audit-level=high`.
4. **Pendiente del owner (configuración GitHub, no código):** añadir el job
   `Security scan` a branch protection, como preveía el plan de cierre.
